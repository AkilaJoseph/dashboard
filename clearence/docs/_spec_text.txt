
TECHNICAL SPECIFICATION
FOR AN AUTOMATED CLEARANCE INFORMATION MANAGEMENT SYSTEM FOR UNIVERSITY STUDENTS
Developed using PHP (Laravel Framework) and MySQL
Version 1.0

1. Introduction
Purpose
This document defines the technical specification for the Automated Clearance Information Management System (ACIMS), a web-based platform designed to digitise and streamline the student clearance process in higher learning institutions. The system shall provide a centralised platform for students, departmental staff, and administrators to initiate, process, monitor, and finalise clearance requests electronically, replacing the manual, paper-based workflow currently employed in most universities.
Scope
The system shall support the following core capabilities:
Online student registration and authentication.
Submission and tracking of clearance requests.
Departmental review, approval, and rejection workflows with comments.
Administrative oversight, user management, and final clearance certification.
Generation of downloadable clearance certificates in PDF format.
Role-based access control for students, departmental staff, and administrators.
Reporting and analytics on clearance status, turnaround time, and departmental performance.
2. System Overview
2.1. Background of the System
Student clearance is a mandatory procedure undertaken before graduation, inter-semester transition, or withdrawal from a university. It requires students to obtain sign-off from multiple departments—including the library, finance office, hostel management, faculty, and dean of students—to confirm that they have no outstanding obligations. In most universities, this process remains manual: students physically move from one office to another carrying a paper clearance form, which is prone to loss, delay, forgery, and bureaucratic bottlenecks.
The Automated Clearance Information Management System (ACIMS) proposes a digital alternative that leverages modern web technologies to automate the end-to-end clearance workflow. The platform shall be built using the PHP Laravel framework with a MySQL relational database, ensuring maintainability, security, and scalability.
2.2. Problem Statement
The existing manual clearance process suffers from the following shortcomings:
Excessive time consumption, with students often requiring several days or weeks to complete clearance.
Loss, damage, or forgery of paper clearance forms.
Absence of a centralised tracking mechanism, making it difficult for students to monitor progress.
Limited visibility for administrators regarding departmental performance and bottlenecks.
High administrative overhead in storing, retrieving, and archiving physical records.
Lack of accountability when clearance officers misplace or delay requests.
Consequently, there is a compelling need for an automated, web-based solution that eliminates these inefficiencies and provides a transparent, verifiable, and auditable clearance process.
2.3. Objectives
2.3.1. General Objective
To design and develop a secure, efficient, and user-friendly web-based Automated Clearance Information Management System that enables university students to request, track, and obtain clearance certificates electronically.
2.3.2. Specific Objectives
To analyse and document the existing manual clearance process and identify its limitations.
To design a relational database schema that captures students, departments, clearance requests, and approvals.
To develop role-based interfaces for students, departmental staff, and administrators using the Laravel framework.
To implement a workflow engine that routes clearance requests sequentially or in parallel across departments.
To integrate PDF generation for clearance certificates and reporting modules for administrators.
To ensure the system enforces robust authentication, authorisation, and data-integrity mechanisms.
To evaluate the system through unit, integration, and user acceptance testing.
3. System Architecture
3.1. Architectural Overview
The system shall adopt a three-tier client–server architecture built upon the Model–View–Controller (MVC) design pattern provided natively by the Laravel framework. The separation of concerns afforded by MVC improves maintainability, testability, and scalability of the codebase.
3.1.1. Presentation Tier (Client)
The presentation tier consists of web pages rendered in the user&apos;s browser. It is built using HTML5, CSS3, JavaScript, and the Blade templating engine, enhanced with Bootstrap for responsive layout and jQuery/Axios for asynchronous interactions.
3.1.2. Application Tier (Server)
The application tier hosts the Laravel application, which encapsulates all business logic, routing, request validation, authentication, and workflow orchestration. It exposes both Blade-rendered views and JSON API endpoints.
3.1.3. Data Tier (Database)
The data tier is a MySQL relational database that persists user accounts, clearance records, departmental metadata, approval histories, and system logs. The application communicates with the database through Laravel&apos;s Eloquent ORM.
3.2. MVC Implementation in Laravel
Models: Eloquent classes (e.g., Student, Department, ClearanceRequest, Approval) that represent database tables and encapsulate data access logic.
Views: Blade templates (.blade.php) that compose HTML presentation with dynamic data.
Controllers: Classes in app/Http/Controllers that receive HTTP requests, delegate to services or models, and return responses.
Routes: Defined in routes/web.php and routes/api.php, mapping URIs to controller actions.
Middleware: Used to enforce authentication, role-based access, CSRF protection, and request throttling.

4. Functional Requirements
This section describes the behavioural requirements of the system, organised by user module. Each requirement is stated in the form &quot;The system shall…&quot; to reflect a verifiable specification.
4.1. Student Module
4.1.1. Registration and Authentication
The system shall allow prospective students to register an account using their registration number, full name, faculty, programme, and institutional email address.
The system shall validate the uniqueness of the registration number and email during account creation.
The system shall require students to set a password that complies with the institution&apos;s password policy (minimum 8 characters, mixed case, numeric, and special character).
The system shall provide secure login using email or registration number and password.
The system shall implement a &quot;Forgot Password&quot; mechanism that sends a time-bounded reset link to the student&apos;s registered email address.
The system shall lock a student account after five consecutive failed login attempts and notify the account holder.
4.1.2. Profile Management
The system shall allow students to view and update personal details such as phone number, postal address, and profile picture.
The system shall restrict modification of academic identifiers (registration number, programme, year of study) to administrators.
4.1.3. Clearance Request Submission
The system shall allow authenticated students to initiate a new clearance request by selecting the clearance type (graduation, inter-semester, withdrawal, transfer).
The system shall automatically attach the list of departments relevant to the selected clearance type.
The system shall allow students to attach supporting documents (e.g., fee receipts, library return slips) in PDF, JPG, or PNG formats not exceeding 5 MB per file.
The system shall generate a unique reference number for each submitted clearance request.
The system shall prevent a student from submitting a new request while another is still in progress.
4.1.4. Clearance Tracking
The system shall provide a dashboard showing the real-time status of the student&apos;s active clearance request.
The system shall display each department&apos;s status as Pending, Approved, or Rejected.
The system shall display any comments or reasons provided by departmental staff during review.
The system shall notify the student via in-app alerts and email whenever the status of any department changes.
4.1.5. Certificate Download
The system shall enable students to download a PDF clearance certificate once all departments have approved and the administrator has granted final approval.
The system shall embed a unique QR code on each certificate that links to an online verification page.
The system shall prevent download of certificates for requests that have not reached the Fully Approved state.
4.2. Department Module
4.2.1. Request Inbox
The system shall present each departmental officer with a dashboard listing all pending clearance requests routed to their department.
The system shall allow filtering of requests by status, date range, clearance type, and student programme.
The system shall allow searching by student registration number or name.
4.2.2. Review and Decision
The system shall display all relevant student information and attached supporting documents for each request.
The system shall allow departmental staff to approve or reject a request.
The system shall require a comment when a request is rejected, stating the reason for rejection.
The system shall permit optional comments when a request is approved.
The system shall record the decision, timestamp, and identity of the approving officer in an immutable approval log.
4.2.3. Departmental Reporting
The system shall provide each department with summary statistics: total requests, approved, rejected, and average turnaround time.
The system shall allow departmental officers to export request lists as CSV or PDF for internal records.
4.3. Administrator Module
4.3.1. User Management
The system shall enable administrators to create, view, edit, deactivate, and delete user accounts.
The system shall allow administrators to reset user passwords and unlock accounts.
The system shall support bulk student import from a CSV file.
4.3.2. Department Management
The system shall allow administrators to create, edit, and deactivate departments.
The system shall allow administrators to assign or unassign departmental officers to a department.
The system shall allow administrators to configure the department routing order for each clearance type.
4.3.3. Final Approval and Certification
The system shall permit administrators to grant final clearance only after all departmental approvals have been obtained.
The system shall digitally sign the final certificate with the institution&apos;s stamp and administrator&apos;s name.
The system shall allow administrators to revoke a previously issued certificate when irregularities are discovered, recording the reason and timestamp.
4.3.4. System Reporting
The system shall generate institutional-level reports, including cleared students per faculty, average clearance duration, and rejection trends.
The system shall support filtering reports by academic year, semester, programme, and department.
The system shall allow exporting reports in PDF and CSV formats.
5. Non-Functional Requirements
5.1. Security
All data in transit shall be encrypted using TLS 1.2 or higher (HTTPS).
User passwords shall be hashed using Laravel&apos;s bcrypt implementation with a work factor of at least 10.
The system shall implement Cross-Site Request Forgery (CSRF) protection on all state-changing requests.
The system shall sanitise and validate all user input to prevent SQL injection and Cross-Site Scripting (XSS).
Session tokens shall expire automatically after 30 minutes of inactivity.
5.2. Performance
The system shall load any page in under 3 seconds under a load of 200 concurrent users on a 10 Mbps connection.
Database queries shall be optimised through appropriate indexing on foreign keys and frequently queried fields.
The system shall employ server-side caching (Redis or file cache) for static reference data such as departments and clearance types.
5.3. Usability
The user interface shall be built upon responsive design principles, supporting viewports from 320 px (mobile) to 1920 px (desktop).
The system shall provide clear, contextual feedback for every user action (e.g., success banners, validation errors).
The interface shall follow consistent navigation patterns and visual hierarchy across all pages.
The system shall provide an inline help section and a downloadable user manual.
5.4. Scalability
The application shall be stateless at the server tier to permit horizontal scaling behind a load balancer.
The database schema shall be normalised to the Third Normal Form (3NF) to support growth without structural refactoring.
The system shall be deployable via Docker containers to facilitate consistent scaling across environments.
5.5. Reliability and Availability
The system shall maintain an operational availability of at least 99% during the academic calendar.
Database backups shall be scheduled daily with a retention period of 30 days.
The system shall log all critical errors and unhandled exceptions for administrative review.
5.6. Maintainability
The source code shall follow the PSR-12 coding standard and Laravel best practices.
All non-trivial classes and methods shall be documented with PHPDoc comments.
The codebase shall be managed using Git with a clear branching strategy.

6.2. Entity Relationships
One users record corresponds to at most one students record (1:1, for student users).
One departments record has many users (1:N, for staff users).
One students record has many clearance_requests (1:N).

One clearance_requests record has many attachments (1:N).
One users record (staff) has many approvals (1:N).
One users record has many notifications (1:N).
7. System Workflow
The clearance process follows a defined sequential and parallel workflow:
7.1. Request Initiation
The student logs into the system using their credentials.
The student selects &quot;Apply for Clearance&quot; from their dashboard.
The student chooses the clearance type (e.g., graduation).
The system displays the list of departments involved and prompts for required attachments.
The student uploads supporting documents and submits the request.
The system creates a clearance_requests record with status = in_progress and seeds one approvals record per department with status = pending.
The system generates a unique reference number and dispatches a confirmation email to the student.
7.2. Departmental Review
Each departmental officer receives an in-app and email notification of a new pending request.
The officer logs in, opens the request, and reviews the student&apos;s details and attachments.
The officer either approves (with optional comment) or rejects (with mandatory reason).
The system updates the corresponding approvals record and notifies the student of the new status.
7.3. Consolidation and Final Approval
Once all departments have approved the request, the system marks the clearance_requests status as awaiting_final.
The administrator receives a notification to perform the final review.
The administrator inspects the full approval chain and grants final clearance, setting status = approved and completed_at = NOW().
7.4. Rejection Handling
If any department rejects the request, the overall status is set to rejected.
The student is notified of the rejection along with the reason provided by the rejecting department.
The student may rectify the issue and submit a fresh clearance request; previous records are retained for audit.
7.5. Certificate Issuance
Upon final approval, the system renders a PDF clearance certificate containing the student&apos;s details, approval chain, institutional seal, administrator&apos;s signature, and a verification QR code.
The student can download the certificate from their dashboard indefinitely.
Third parties can scan the QR code to verify the certificate&apos;s authenticity on a public verification page.

11. Security Considerations
11.1. Authentication and Authorisation
The system shall use Laravel&apos;s built-in authentication scaffolding (Laravel Breeze / Fortify) with session-based authentication for web routes and Sanctum tokens for API routes.
Passwords shall be hashed using bcrypt and never stored or transmitted in plain text.
Role-based access control shall be enforced through Laravel Gates, Policies, and role-checking middleware.
The system shall support optional two-factor authentication for administrative accounts.
11.2. Data Validation
All incoming requests shall be validated using Laravel Form Requests.
File uploads shall be validated for MIME type, extension, and size before persistence.
Uploaded files shall be stored outside the public web root and served through authenticated controller actions.
11.3. Protection Against Common Attacks
SQL Injection: prevented by exclusive use of Eloquent ORM and parameterised queries.
Cross-Site Scripting (XSS): prevented by default escaping of Blade output and sanitisation of user input.
Cross-Site Request Forgery (CSRF): mitigated by Laravel&apos;s CSRF token middleware on all state-changing forms.
Session Hijacking: mitigated by HTTPS, HttpOnly and Secure cookie flags, and session regeneration on login.
Brute Force Attacks: mitigated by rate limiting on the login endpoint and account lockout after repeated failures.
Mass Assignment: prevented through explicit $fillable and $guarded declarations on Eloquent models.
Sensitive Data Exposure: addressed by HTTPS, environment-scoped secrets in .env, and encryption of confidential fields with Laravel&apos;s Crypt facade.
12. Testing Strategy
Testing shall be conducted at multiple levels to verify both correctness and fitness for purpose.
12.1. Unit Testing
Conducted using PHPUnit, shipped with Laravel.
Tests shall cover individual model methods, validation rules, service classes, and policy logic.
A code-coverage threshold of at least 70 % shall be targeted for critical business logic.
12.2. Feature (Integration) Testing
Conducted using Laravel&apos;s HTTP test helpers to simulate requests end-to-end.
Scenarios shall include successful clearance submission, departmental approval, rejection, and final certification.
Tests shall use an isolated SQLite or MySQL test database seeded with fixtures.
12.3. System Testing
Conducted on a staging environment that mirrors production.
Includes functional, performance, and compatibility testing across major browsers (Chrome, Firefox, Edge, Safari) and devices.
Performance testing shall be carried out using tools such as Apache JMeter to simulate concurrent users.
12.4. User Acceptance Testing (UAT)
Conducted with representative users from each role: students, departmental officers, and administrators.
Scripted test scenarios shall align with the functional requirements.
Findings shall be documented in a UAT report, and identified defects addressed before final delivery.
12.5. Security Testing
Includes vulnerability scanning (e.g., using OWASP ZAP) and manual penetration testing for OWASP Top 10 vulnerabilities.
Results shall be remediated prior to production deployment.

13.2. Installation Steps
Provision the server and install all prerequisite software listed above.
Clone the project repository from the Git remote into /var/www/acims.
Run composer install --no-dev --optimize-autoloader to install PHP dependencies.
Run npm install &amp;&amp; npm run build to compile front-end assets.
Copy .env.example to .env and configure database credentials, mail settings, and application URL.
Run php artisan key:generate to create the application encryption key.
Create the MySQL database and user, and run php artisan migrate --seed to build the schema and seed reference data.
Set appropriate filesystem permissions on storage and bootstrap/cache directories.
Configure the web server virtual host to point to /var/www/acims/public.
Install and configure the SSL certificate to enforce HTTPS.
Schedule php artisan schedule:run via cron every minute to handle background tasks such as notifications.
Verify deployment by accessing the application URL and performing a smoke test with a seeded administrator account.
